根据你的要求，我将对项目架构进行重新规划。**数据部分保持不变**（底层驱动、数据结构定义、数据访问层依然独立且通用），而**缓存层和复杂查询逻辑层不再作为全局系统组件**，改为与特定 UI 页面写在一起，缓存相关的类型定义统一放在 `types.ts` 中。每个页面根据自己的业务需求，针对性地实现所需缓存和查询逻辑，从而实现更高内聚、更低耦合的页面级架构。

以下是新的项目层次结构说明书。

---

## 项目层次结构说明书（页面内聚版）

### 一、总体架构思想

- **数据层保持独立**：IndexedDB 驱动、数据结构定义、数据访问层（Repository）为通用基础，供所有页面复用。
- **缓存与查询逻辑下沉至页面**：不再提供全局的 `cache` 和 `query` 模块，每个页面（或功能模块）根据自身需要，在页面内部实现自己的缓存策略和查询/过滤/排序逻辑。
- **缓存类型定义统一**：所有与缓存相关的 TypeScript 类型（如缓存记录结构、引用计数类型等）统一放在 `src/types.ts`（或各模块共享的 `types.ts`）中，便于复用和类型检查。
- **页面间不共享缓存**：除非需要（如使用全局状态管理），否则每个页面独立管理自己的缓存数据，避免全局缓存带来的生命周期复杂度和内存泄漏风险。

---

### 二、各层职责与能力边界

#### 1️⃣ `src/database/indexeddb`（底层驱动层）

**保持不变**  
封装 IndexedDB 原生 API，提供通用 CRUD、索引查询、游标迭代等基础能力。  
**禁止任何业务逻辑**。

#### 2️⃣ `src/database/types`（数据结构层）

**保持不变**  
定义所有数据实体类型（如 `Creator`、`CreatorAsset`），不包含任何方法或业务逻辑。

#### 3️⃣ `src/database/implementations`（数据访问层）

**保持不变**  
针对每个表（如 `CreatorRepository`、`CreatorAssetRepository`）提供最小查询方法（`getById`、`getByIds`、`getByPlatform` 等），支持部分字段加载。  
**禁止复杂查询、排序、多条件组合**。  
重数据（如 `CreatorAsset`）按需通过独立方法获取。


---

#### 5️⃣ 页面层（UI + 缓存 + 查询逻辑）

每个页面（或功能模块）内部包含以下三个部分，写在同一目录或同一文件中，实现高内聚：

- **页面组件**（UI 渲染与交互）
- **页面内缓存模块**（管理该页面所需数据的缓存，可能包括轻数据全量缓存、重数据按需缓存）
- **页面内查询逻辑**（对缓存数据进行过滤、排序、分页等复杂操作）

**目录结构示例**：

```
src/UI/CreatorList/
├── CreatorList.ts          // UI核心代码
├── Query.ts      // 页面内查询/过滤/排序逻辑
└── types.ts                 // 页面内私有类型,同时定义了需要实现复杂查询过滤功能的缓存结构
```

**页面内缓存模块职责**：

- 从 `implementations` 层获取原始数据（轻数据或重数据）
- 将数据存储在内存中（例如 `Map`、`WeakMap` 或简单对象）
- 提供获取、更新、清除缓存的 API
- 不跨页面共享

**页面内查询逻辑职责**：

- 从本页面的缓存模块获取数据
- 在内存中实现筛选、排序、分页等复杂操作（纯内存计算）
- 返回处理后的结果供 UI 渲染
- 不直接访问 IndexedDB 或 Repository

**页面组件职责**：

- 调用页面内查询逻辑获取展示数据
- 触发重数据加载（如头像、详情）时，调用页面内缓存模块按需获取并缓存
- 处理加载状态、错误边界

---

### 三、轻重数据分离规则（与之前一致）

| 数据类型 | 缓存策略 | 生命周期 | 加载时机 |
|----------|----------|----------|----------|
| 轻数据   | 页面内全量缓存 | 页面打开 → 页面卸载时清除 | 页面初始化时全量或分页加载 |
| 重数据   | 页面内按需缓存（LRU 或简单 Map） | 页面可见期间，随 UI 触发加载 | 视口进入时、详情页进入时 |

判定标准：单个字段超过 10KB 或为 base64/blob、长文本等视为重数据。

---

### 四、运行机制（页面内聚版）

以“创作者列表页”为例：

1. **页面初始化**
   - 页面组件调用 `creatorListCache.loadLightCreators()`，该方法通过 `CreatorRepository.getPartial(['id', 'name', 'avatarUrl'])` 获取轻数据列表。
   - 轻数据缓存到 `creatorListCache.lightMap` 中。

2. **用户筛选/排序**
   - 页面组件调用 `creatorListQuery.filterByTag(tag)`，该函数内部从缓存中读取轻数据，进行内存过滤，返回结果。
   - 排序、分页同样在 `query` 模块中完成。

3. **重数据按需加载**
   - 列表项进入视口时，调用 `creatorListCache.loadAvatar(creatorId)`，该方法通过 `CreatorAssetRepository.getByCreatorId(creatorId)` 获取完整头像数据，并缓存在页面内缓存中。
   - 详情页打开时，调用 `creatorListCache.loadFullCreator(creatorId)` 加载完整描述等重数据。

4. **页面卸载**
   - 页面组件卸载时，清除页面内所有缓存（如 `clear()`），释放内存。

### 六、严格执行约束（更新版）

1. **数据层保持独立**  
   `indexeddb`、`types`、`implementations` 仅提供数据存取能力，不包含任何页面特定逻辑。

2. **页面内缓存和查询逻辑不能跨页面共享**  
   每个页面独立管理自己的缓存，如需跨页面共享数据，应使用全局状态管理（如 Zustand、Redux）并谨慎设计生命周期。

3. **缓存类型定义统一放在 `每个模块/types.ts`**  
   所有页面共用的缓存相关类型（如 `LightCacheItem`、`HeavyCacheItem`）必须在此处定义，避免重复。

4. **页面内禁止直接调用 `implementations` 或 `indexeddb` 进行复杂查询**  
   所有复杂操作（过滤、排序、组合条件）必须在页面内的查询模块中完成，不能写在 Repository 或底层驱动中。

5. **重数据必须按需加载**  
   页面内缓存模块应提供按需获取重数据的方法，不得在页面初始化时全量加载重数据。

6. **页面卸载时必须清理缓存**  
   每个页面应在组件卸载时调用缓存清理函数，防止内存泄漏。

---

### 七、一句话架构总结（更新版）

**“数据层独立通用，缓存与查询逻辑随页面内聚，按需实现，类型定义统一。”**
