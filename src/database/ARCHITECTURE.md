# 数据层架构设计

## 目录结构

### 1. 基础数据层
- `src/database/types/` - 唯一数据定义，保持不变，所有的基础
- `src/database/indexeddb/` - 持久化数据库提供的底层支持
- `src/database/implementations/` - 对数据库不同表数据实现的操作封装，对外开放数据操作

### 2. 内存查询层
- `src/database/cache/` - 基于内存实现的索引和数据缓存模型
- `src/database/manager/` - 管理内存中数据和生命周期的模块
- `src/database/query/` - 基于cache的高性能复杂查询纯函数实现
- `src/database/repository/` - 基于上述模块实现的高性能复杂查询业务的对外接口

## 模块职责

### 基础数据层
1. **types** - 定义所有数据类型，是整个数据层的基础
2. **indexeddb** - 提供持久化数据库的底层支持，包括数据库管理、事务处理等
3. **implementations** - 封装对数据库的直接操作，为系统其他模块提供数据访问能力

### 内存查询层
1. **cache** - 实现内存中的数据缓存和索引，提高查询性能
2. **manager** - 管理内存中的数据和生命周期，包括缓存更新、数据同步等
3. **query** - 提供基于内存缓存的高性能复杂查询功能，纯函数实现
4. **query-service** - 实现复杂查询功能的模块,用于补全功能

## 两个入口点的区别与用途

### Repository层入口

**用途**：
- UI交互场景下的数据查询和操作
- 基于缓存和策略的复杂查询
- 需要分页、过滤、排序等功能的场景

**特点**：
- 薄层设计，只负责将UI请求转为数据请求描述
- 包含缓存机制，提高查询性能
- 支持复杂查询策略
- 自动管理数据一致性

**使用场景**：
- 列表页数据加载
- 搜索功能
- 数据筛选和排序
- 单个对象的查询和更新
